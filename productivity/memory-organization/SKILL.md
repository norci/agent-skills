# 记忆整理之律

## 记忆分储原则

记忆存三处，各司其职：

- **MEMORY.md**：系统自动加载之规则，约束一切会话的行为准则与索引
- **USER.md**：君之好恶、风格、工具偏好
- **custom-skills**：专用流程，以技能形式存放

**同一事实不存二处。** 整理前先辨条目归属

### 归属判准

- 涉言行准则、应对、发问、君臣之礼 → MEMORY.md
- 涉用户个人偏好（语言风格、工具偏好） → USER.md
- 涉专用流程、操作指南 → custom-skills

若有疑，宁存问君，毋妄归

## 技能管理之律

### 输出目录

自动生成的技能输出至 `~/.hermes/custom-skills/`。此目录已在 config.yaml 的 `skills.external_dirs` 中注册，重启后生效

```yaml
skills:
  external_dirs:
    - ~/.hermes/custom-skills
```

### 来源鉴别

被问及技能来源时，须依次核查三个权威数据源，不得凭目录位置或印象作答：

- `.bundled_manifest`：`~/.hermes/skills/.bundled_manifest` —— 列表中的为 Hermes 内置技能
- `.hub/lock.json`：`~/.hermes/skills/.hub/lock.json`，检查 `installed` 字段 —— hub 安装的为社区技能
- 可选技能目录：`/home/josephx/.hermes/hermes-agent/optional-skills/`

三者皆否，则为自创。**以数据源为准，不以文件位置或个人记忆为凭** —— 技能目录可被手动移动，故不可依赖目录枚举

## 触发条件

- 当用户要求整理、合并或清理记忆条目，发现重复或语义重叠时
- 当用户提供新的规则、箴言、行为守则并要求整合进现有记忆时
- 当用户指出当前记忆条目结构混乱、层级不清或内容过度简化时

## 操作陷阱

### `memory replace` 乃整条替换，非节选修改

`memory(action='replace', old_text='...', content='...')` 的行为：
- `old_text` 匹配到某一条完整记忆条目后，**将该条目整体替换**为 `content`，而非仅替换匹配到的那一小节
- 若目标条目内含多个独立章节，但只想修改其中一个小节，**不可只传待改小节作 `content`** —— 否则其他章节尽失

**正确做法**：
- 读取该条目完整内容
- 在完整内容基础上，仅修改目标小节
- 将**完整修改后的内容**传入 `content`
- 用唯一性足够的 `old_text` 定位

**补救**：若已误替换导致丢失章节，须凭记忆或此前会话记录完整还原后重新 replace

### 新增规则整合之法

当用户提供新的规则、箴言、行为守则要求整合进现有记忆时，遵循以下流程：

**通读全篇，不可割裂**

接受新规则后，先通读**现有记忆的全部条目**，同时通读用户提供的新规则全文。不能只看待改小节

**逐句比对，标记交叉**

将新规则的每一条与记忆中的每一条做交叉对比。关注三类关系：
- **完全重叠**：同一事换说法说两次 → 应合并
- **部分交叉**：两律边界有交集但核心不同 → 精确定义边界，各自保留
- **互补关系**：一为底限（不能做什么），一为上限（应该做什么） → 同一体系的两面，应共存

特别留意：答问守则、答问箴、应对四戒 —— 三者同一体系不同维度，不可轻率合并

**提出结构方案**

在整合前，先向用户提出结构重组方案，说明：
- 哪些条目与新规则有交叉
- 建议如何重新组织（合并、拆出、新立、保留）
- 每条拟建条目的边界定义

待用户认可后方执行。不可擅自重构

**执行整合，保留原意**
- 整合时**不可过度简化**以致丢失原意。宁可稍繁，不可失准
- 每条核心意涵须保留，不可因简化而隐没其理
- 每个典型示例须保留，不可因简化而失其代表性

**验证条理**

整合完成后，自检：
- 每条独立、边界清晰
- 无语义重复
- 无遗漏核心内容
- 语言风格统一为典雅文言
- 层级分明（总纲→节→条）

## 操作步骤

- **读取文件**：读取 MEMORY.md 与 USER.md 的完整内容
- **提取所有条目**：遍历文件提取全部条目
- **辨明归属**：按分储原则，将误置条目移回正确位置
- **辨析重叠**：
  - **完全重叠** → 合并
  - **部分交叉** → 精确定义边界，保留各自独立
  - **时序互补** → 保留为独立条目
- **边界定义法**：对每对疑似重叠条目，回答三个问题：
  - **何时触**
  - **管何事**
  - **有何别**
  若三问均有明确答案，则两律应**各自保留**
- **合并策略**：
  - 仅对完全重叠的条目合并
  - 合并后的条目名称应涵盖原有核心意义
  - 语言须精炼
- **覆写文件**：将重整后的内容写入对应文件
- **验证**：
  - 无语义重复条目
  - 无交叉归属
  - 每条独立、边界清晰
  - 语言风格统一为雅言

## 辨析陷阱

### 「记述之律」vs「译名之律」

- 记述之律管「写什么、怎么写」 —— Skill/Memory 条目的文风
- 译名之律管「翻译什么」 —— 输出中技术专名不译
- 交叉处：皆涉技术名词保留原貌 —— 此为合理共存

### 「验证之律」vs「查错之律」vs「推诿之戒」

- 验证之律管「改完之后」 —— 自验其正
- 查错之律管「出了问题」 —— 溯其源
- 推诿之戒管「犯错时之态度」 —— 不推诿、速改策
- 三者时序不同（事前→事中→事后），应各自保留

### Skill 与 MEMORY.md 之不可引用律

Skill 文件中毋引用 MEMORY.md 或以「此篇不赘述，以 MEMORY.md 为准」之类措辞转嫁内容。MEMORY.md 每次会话自动注入 system prompt，skill 加载时不可预设其已存在。二处内容当各自完整独立

**实操原则**：
- **通用准则**（服务之心、真理之志、应对之纲、言行之界、辅助之心、根本之诫、模型偏见之诫、工具之律、言辞之度、润色之律） → 全归 MEMORY.md
- **类型特有**（场景识别、特定处置流程、因用户实践亲定之规则） → 全归 skill
- 判定标准：此条若适用于一切会话，入 MEMORY.md；若仅在触发该 skill 的场景中生效，留 skill

## 注意事项

- **语言风格**：所有条目须用雅言，禁用白话和英文（技术名词除外）
- **技术名词**：`uv`、`Venv`、`Repo`、`pip`、`clarify` 等须保留原貌
- **辨析优先于合并**：若存疑，保留之

## 验证标准

- 无语义重复条目
- 无交叉归属
- 每条独立、边界清晰
- 语言风格统一为雅言
